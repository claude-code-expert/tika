# Tika 알림 기능 체크리스트

> 수동 QA 체크리스트 — 인앱 알림 트리거, UI, 설정 항목

---

## 1. 알림 UI

### 1-1. 헤더 벨 아이콘

- [ ] 헤더에 알림 벨 아이콘이 표시된다
- [ ] 읽지 않은 알림이 1개 이상이면 벨 아이콘에 숫자 배지가 표시된다
- [ ] 읽지 않은 알림이 없으면 배지가 표시되지 않거나 0이 표시되지 않는다
- [ ] 배지 숫자는 최대 99까지 표시되고 초과 시 "99+" 형식으로 표시된다

### 1-2. 알림 드롭다운 / 패널

- [ ] 벨 아이콘 클릭 시 알림 목록 드롭다운이 열린다
- [ ] 드롭다운에 최근 알림 목록이 시간 역순으로 표시된다
- [ ] 각 알림 항목에 제목(title)과 메시지(message)가 표시된다
- [ ] 각 알림 항목에 발생 시각이 표시된다 (예: "5분 전", "2시간 전")
- [ ] 읽지 않은 알림은 읽은 알림과 시각적으로 구분된다 (배경색 등)
- [ ] 알림 항목 클릭 시 해당 티켓/워크스페이스로 이동한다
- [ ] 알림 항목 클릭 시 읽음 처리된다
- [ ] "모두 읽음 처리" 버튼 클릭 시 모든 알림이 읽음 상태가 된다
- [ ] "모두 읽음 처리" 후 배지 카운터가 0으로 초기화된다

### 1-3. 알림 내역 페이지

- [ ] `/notifications` 페이지에서 전체 알림 내역이 표시된다
- [ ] 알림이 없을 때 빈 상태 메시지가 표시된다
- [ ] 페이지네이션 또는 무한 스크롤로 이전 알림을 조회할 수 있다
- [ ] 각 알림의 워크스페이스 정보가 표시된다

## 2. 알림 트리거 — 14개 유형

### 2-1. TICKET_STATUS_CHANGED (티켓 상태 변경)

- [ ] 티켓 상태 변경 시 (BACKLOG → TODO → IN_PROGRESS → DONE 등) 담당자에게 알림이 발송된다
- [ ] 상태 변경자 본인에게는 알림이 발송되지 않는다
- [ ] 알림 제목: "티켓 상태 변경" 유형의 제목이 포함된다
- [ ] 알림 메시지: 변경된 상태 정보가 포함된다

### 2-2. TICKET_COMMENTED (티켓 댓글)

- [ ] 티켓에 새 댓글 작성 시 담당자/관련자에게 알림이 발송된다
- [ ] 댓글 작성자 본인에게는 알림이 발송되지 않는다
- [ ] 알림 메시지: 어떤 티켓에 댓글이 달렸는지 정보가 포함된다

### 2-3. TICKET_ASSIGNED (담당자 지정)

- [ ] 티켓에 담당자로 지정될 때 해당 사용자에게 알림이 발송된다
- [ ] 지정자 본인이 자기 자신을 담당자로 지정할 때는 알림이 발송되지 않는다
- [ ] 알림 메시지: 어떤 티켓의 담당자로 지정되었는지 정보가 포함된다

### 2-4. TICKET_UNASSIGNED (담당자 해제)

- [ ] 티켓 담당자에서 제거될 때 해당 사용자에게 알림이 발송된다
- [ ] 알림 메시지: 어떤 티켓의 담당자에서 해제되었는지 정보가 포함된다

### 2-5. TICKET_DELETED (티켓 삭제)

- [ ] 티켓이 삭제(휴지통 이동)될 때 담당자에게 알림이 발송된다
- [ ] 삭제자 본인에게는 알림이 발송되지 않는다
- [ ] 알림 메시지: 삭제된 티켓 제목이 포함된다

### 2-6. DEADLINE_WARNING (마감일 D-1 경고)

- [ ] 마감일 하루 전(D-1) 00:00 UTC에 크론 잡이 실행된다
- [ ] 내일 마감인 티켓의 담당자에게 알림이 발송된다
- [ ] 담당자가 없는 티켓은 인앱 알림이 발송되지 않는다 (외부 채널만)
- [ ] Slack 채널이 활성화된 경우 Slack 메시지도 발송된다
- [ ] `GET /api/cron/notify-due` 요청 시 `Authorization: Bearer {CRON_SECRET}` 헤더가 필요하다
- [ ] 잘못된 CRON_SECRET으로 요청 시 `401` 에러가 반환된다
- [ ] 발송 성공 시 `notificationLogs`에 `SENT` 상태로 기록된다
- [ ] 발송 실패 시 `notificationLogs`에 `FAILED` 상태로 기록된다
- [ ] 응답 바디에 `{ processed, sent, failed }` 통계가 포함된다

### 2-7. INVITE_RECEIVED (초대 수신)

- [ ] 초대 링크를 통해 워크스페이스에 가입하면 초대한 사람에게 알림이 발송된다
- [ ] 알림 메시지: 누가 초대를 수락했는지 정보가 포함된다
  - (현재 구현 상태 확인 필요 — 초대 수락 API에서 알림 트리거 연결 여부)

### 2-8. ROLE_CHANGED (역할 변경)

- [ ] OWNER가 멤버의 역할을 변경할 때 해당 멤버에게 알림이 발송된다
- [ ] 알림 메시지: 변경된 역할 정보가 포함된다

### 2-9. MEMBER_JOINED (멤버 가입)

- [ ] 새 멤버가 워크스페이스에 가입할 때 기존 멤버들에게 알림이 발송된다
- [ ] 가입자 본인에게는 발송되지 않는다
- [ ] 알림 메시지: 누가 워크스페이스에 가입했는지 정보가 포함된다

### 2-10. MEMBER_REMOVED (멤버 제거)

- [ ] OWNER가 멤버를 제거할 때 제거된 멤버에게 알림이 발송된다
- [ ] 알림 메시지: 어떤 워크스페이스에서 제거되었는지 정보가 포함된다

### 2-11. JOIN_REQUEST_RECEIVED (참여 신청 수신)

- [ ] 사용자가 팀 워크스페이스에 참여 신청을 하면 워크스페이스 OWNER에게 알림이 발송된다
- [ ] 알림 메시지: 누가 신청했는지 정보가 포함된다

### 2-12. JOIN_REQUEST_RESOLVED (참여 신청 처리 완료)

- [ ] OWNER가 참여 신청을 승인할 때 신청자에게 "승인" 알림이 발송된다
- [ ] OWNER가 참여 신청을 거절할 때 신청자에게 "거절" 알림이 발송된다
- [ ] 알림 메시지: 신청 처리 결과(승인/거절)가 포함된다

### 2-13. SPRINT_STARTED (스프린트 시작)

- [ ] OWNER가 스프린트를 활성화하면 워크스페이스 모든 멤버에게 알림이 발송된다
- [ ] 알림 메시지: 어떤 스프린트가 시작되었는지 정보가 포함된다

### 2-14. SPRINT_COMPLETED (스프린트 완료)

- [ ] OWNER가 스프린트를 완료하면 워크스페이스 모든 멤버에게 알림이 발송된다
- [ ] 알림 메시지: 어떤 스프린트가 완료되었는지 정보가 포함된다

## 3. 알림 설정 (Notification Preferences)

> 설정 경로: `/settings` → 알림 설정 탭

### 3-1. 카테고리 및 토글 항목

#### 티켓 알림 카테고리

- [ ] TICKET_ASSIGNED 토글이 표시된다 (기본값: 활성화)
- [ ] TICKET_UNASSIGNED 토글이 표시된다 (기본값: 활성화)
- [ ] TICKET_STATUS_CHANGED 토글이 표시된다 (기본값: 활성화)
- [ ] TICKET_COMMENTED 토글이 표시된다 (기본값: 활성화)
- [ ] TICKET_DELETED 토글이 표시된다 (기본값: 활성화)
- [ ] DEADLINE_WARNING 토글이 표시된다 (기본값: 활성화)

#### 워크스페이스 알림 카테고리

- [ ] MEMBER_JOINED 토글이 표시된다 (기본값: 활성화)
- [ ] MEMBER_REMOVED 토글이 표시된다 (기본값: 활성화)
- [ ] ROLE_CHANGED 토글이 표시된다 (기본값: 활성화)
- [ ] JOIN_REQUEST_RECEIVED 토글이 표시된다 (기본값: 활성화)
- [ ] JOIN_REQUEST_RESOLVED 토글이 표시된다 (기본값: 활성화)

#### 스프린트 알림 카테고리

- [ ] SPRINT_STARTED 토글이 표시된다 (기본값: 활성화)
- [ ] SPRINT_COMPLETED 토글이 표시된다 (기본값: 활성화)

### 3-2. 토글 기능

- [ ] 각 토글을 비활성화하면 해당 유형의 알림이 더 이상 발송되지 않는다
- [ ] 비활성화된 토글을 다시 활성화하면 해당 유형의 알림이 다시 발송된다
- [ ] 설정 저장 후 새로고침해도 설정 상태가 유지된다
- [ ] 설정은 사용자별로 독립적으로 저장된다 (다른 사용자 설정에 영향 없음)

### 3-3. 알림 설정 API

- [ ] `GET /api/notifications/preferences` — 현재 설정을 조회한다 (200)
- [ ] `PUT /api/notifications/preferences` — 설정을 저장한다 (200)
- [ ] 인증 없이 요청 시 `401` 에러가 반환된다

## 4. 인앱 알림 API

### 4-1. 알림 조회

- [ ] `GET /api/notifications/in-app` — 내 알림 목록을 반환한다 (200)
- [ ] 인증 없이 요청 시 `401` 에러가 반환된다
- [ ] 응답에 읽음 여부(`isRead`) 필드가 포함된다

### 4-2. 읽음 처리

- [ ] `PATCH /api/notifications/in-app/[id]` — 특정 알림을 읽음 처리한다 (200)
- [ ] `PATCH /api/notifications/in-app` (모두 읽음) — 모든 알림을 읽음 처리한다
- [ ] 다른 사용자의 알림을 읽음 처리 시도 시 에러가 반환된다

---

## 5. 알림 전달 방식 (Fire-and-Forget)

- [ ] 알림 발송 실패 시 API 응답 자체는 실패하지 않는다 (fire-and-forget 패턴)
- [ ] 알림 발송 오류는 서버 로그에 기록된다
- [ ] 알림이 실제 수신자에게 전달되는 것을 DB에서 확인한다 (`inAppNotifications` 테이블)
