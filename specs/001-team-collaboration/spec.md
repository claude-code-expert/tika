# Feature Specification: Team Collaboration Workspace

**Feature Branch**: `001-team-collaboration`
**Created**: 2026-03-04
**Status**: Draft
**Input**: Phase 4 Team — 팀 워크스페이스 협업 기능 구현. RBAC(OWNER/MEMBER/VIEWER) 기반 권한 관리, 멤버 초대 시스템, 다중 담당자, 스프린트 관리, 팀 대시보드·WBS·분석·번다운 5개 화면 제공.

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 팀 워크스페이스 개설 및 멤버 초대 (Priority: P1)

팀장(OWNER)이 팀 워크스페이스를 개설하고, 동료를 이메일로 초대하여 함께 티켓을 관리한다.
현재 개인 워크스페이스를 사용하던 사용자가 팀 협업이 필요할 때 팀 워크스페이스로 전환하거나 신규 개설한다.

**Why this priority**: 팀 협업의 출발점. 워크스페이스와 멤버 관리가 없으면 다른 팀 기능 전체가 동작하지 않는다.

**Independent Test**: 워크스페이스 생성 → 초대 링크 발행 → 초대 수락 → 멤버 목록 확인 흐름만으로 독립적으로 검증 가능하며, 이 단계만으로도 팀 보드 접근이 가능한 MVP를 제공한다.

**Acceptance Scenarios**:

1. **Given** 로그인한 사용자가 팀 워크스페이스가 없을 때, **When** "팀 워크스페이스 만들기"를 클릭하고 이름을 입력하면, **Then** 새 팀 워크스페이스가 생성되고 해당 사용자는 OWNER 역할로 자동 등록된다.
2. **Given** OWNER가 팀 워크스페이스에서 초대를 생성할 때, **When** 이메일 주소와 역할(MEMBER/VIEWER)을 선택하여 초대하면, **Then** 7일 유효기간의 고유 초대 링크가 생성되고 초대 상태는 PENDING이 된다.
3. **Given** 초대 링크를 받은 사용자가 링크를 열면, **When** 초대 미리보기(워크스페이스명·초대자·역할 표시) 페이지에서 "로그인하여 수락"을 클릭하면, **Then** Google 로그인 후 초대 수락 화면으로 자동 복귀되고 "수락" 클릭 시 해당 역할로 멤버가 된다.
4. **Given** 초대 링크가 7일을 초과했을 때, **When** 사용자가 해당 링크를 열면, **Then** "초대가 만료되었습니다" 메시지를 보여주고 새 초대를 요청하도록 안내한다.
5. **Given** OWNER가 멤버를 제거하려 할 때, **When** 해당 멤버를 선택하고 "제거"를 클릭하면, **Then** 멤버가 워크스페이스에서 제거되고 해당 워크스페이스 접근이 차단된다.

### User Story 2 - 역할 기반 접근 제어 (Priority: P1)

각 멤버는 역할(OWNER/MEMBER/VIEWER)에 따라 서로 다른 권한을 갖는다. VIEWER는 읽기만 가능하고, MEMBER는 티켓을 생성·수정할 수 있으며, OWNER는 워크스페이스 설정을 포함한 모든 권한을 갖는다.

**Why this priority**: 권한 제어가 없으면 팀 데이터가 보호되지 않는다. P1 User Story 1과 동시에 구현해야 의미 있는 팀 기능이 된다.

**Independent Test**: VIEWER 계정으로 로그인하여 티켓 생성 시도 → 거부 확인, MEMBER 계정으로 티켓 생성 성공 → 워크스페이스 설정 접근 시도 → 거부 확인의 흐름으로 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** VIEWER 역할의 멤버가 팀 보드에 접근하면, **When** 티켓 생성 버튼을 클릭하려 하면, **Then** 버튼이 비활성화되거나 "권한이 없습니다" 안내가 표시된다.
2. **Given** MEMBER 역할의 멤버가 팀 보드에서, **When** 티켓을 생성·수정·삭제하면, **Then** 작업이 정상적으로 처리된다.
3. **Given** MEMBER가 워크스페이스 설정 페이지에 접근을 시도할 때, **When** URL을 직접 입력하더라도, **Then** 접근이 차단되고 403 응답을 받는다.
4. **Given** OWNER가 멤버의 역할을 변경하면, **When** MEMBER를 VIEWER로 변경하면, **Then** 즉시 해당 멤버의 쓰기 권한이 제거된다.
5. **Given** OWNER 본인이 워크스페이스를 삭제하려 할 때, **When** 워크스페이스 이름을 재입력하고 확인하면, **Then** 워크스페이스와 모든 하위 데이터(티켓, 라벨, 댓글 등)가 삭제된다.

### User Story 3 - 스프린트 계획 및 실행 (Priority: P2)

OWNER가 스프린트를 생성하고, 티켓을 스프린트에 배정하며, 스프린트를 시작·완료하는 전체 사이클을 관리한다.

**Why this priority**: 스프린트는 팀 협업의 핵심 업무 단위이다. 워크스페이스와 RBAC이 완성된 뒤에 독립적으로 구현 가능하다.

**Independent Test**: 스프린트 생성 → 티켓 배정 → 스프린트 시작(ACTIVE) → 완료(COMPLETED) 흐름으로 독립 검증 가능.

**Acceptance Scenarios**:

1. **Given** OWNER가 팀 대시보드에서, **When** 스프린트 이름, 목표, 시작일, 종료일을 입력하고 생성하면, **Then** PLANNED 상태의 스프린트가 목록에 표시된다.
2. **Given** PLANNED 상태의 스프린트가 있을 때, **When** OWNER가 "스프린트 시작"을 클릭하면, **Then** 해당 스프린트가 ACTIVE 상태가 되고 이미 ACTIVE 스프린트가 있다면 오류를 표시한다.
3. **Given** ACTIVE 스프린트가 진행 중일 때, **When** OWNER가 "스프린트 완료"를 클릭하면, **Then** 스프린트가 COMPLETED 상태가 되고 미완료 티켓을 다음 스프린트로 이동할지 묻는 안내가 표시된다.
4. **Given** 스프린트가 존재할 때, **When** 티켓 생성·수정 화면에서 스프린트를 선택하면, **Then** 해당 티켓이 스프린트에 배정된다.
5. **Given** 워크스페이스에 ACTIVE 스프린트가 없을 때, **When** 새 스프린트를 시작하려 하면, **Then** 정상적으로 ACTIVE 상태로 전환된다.

### User Story 4 - 다중 담당자 배정 (Priority: P2)

하나의 티켓에 여러 팀원을 담당자로 배정할 수 있다. 티켓 카드에 담당자 아바타가 표시되며, "내 티켓" 필터로 본인이 담당한 티켓만 볼 수 있다.

**Why this priority**: 팀 협업에서 공동 담당자가 필요한 케이스가 빈번하다. 워크스페이스·RBAC이 선행되어야 하지만 스프린트와는 독립적으로 구현 가능하다.

**Independent Test**: 티켓 생성 시 2명 이상의 담당자를 선택 → 티켓 카드에 아바타 스택 표시 → "내 티켓" 필터 적용 → 본인 담당 티켓만 표시 흐름으로 검증 가능.

**Acceptance Scenarios**:

1. **Given** MEMBER가 티켓을 생성 또는 수정할 때, **When** 워크스페이스 멤버 목록에서 최대 5명을 선택하면, **Then** 선택된 멤버 모두가 담당자로 배정된다.
2. **Given** 다중 담당자가 배정된 티켓 카드에서, **When** 보드나 목록에서 해당 카드를 보면, **Then** 담당자 아바타가 겹쳐(스택) 표시되고 5명 초과 시 "+N" 형식으로 나타난다.
3. **Given** 사용자가 "내 티켓" 필터를 활성화하면, **When** 보드를 조회하면, **Then** 본인이 담당자로 포함된 티켓만 표시된다.
4. **Given** 티켓에 6번째 담당자를 추가하려 할 때, **When** 6번째 멤버를 선택하면, **Then** "담당자는 최대 5명까지 배정할 수 있습니다" 메시지가 표시된다.

### User Story 5 - 팀 대시보드 및 분석 (Priority: P3)

팀의 업무 현황을 한눈에 파악할 수 있는 대시보드와 분석 화면을 제공한다. 번다운 차트, 누적 흐름 다이어그램, 벨로시티, 사이클 타임, 멤버별 워크로드 등을 시각화한다.

**Why this priority**: 데이터 시각화는 실제 업무 데이터가 축적된 이후에 의미가 있으며, 앞선 기능들이 완성된 후에 독립적으로 추가 가능하다.

**Independent Test**: 스프린트와 티켓 데이터가 있는 상태에서 팀 대시보드 접속 → 번다운 차트 렌더링 확인 → 날짜 범위 필터 변경 → 차트 갱신 확인으로 검증 가능.

**Acceptance Scenarios**:

1. **Given** 팀 워크스페이스에 티켓과 스프린트 데이터가 있을 때, **When** 팀 대시보드에 접속하면, **Then** 진행 중 스프린트 현황, Goal 진척율, 멤버별 업무 분포, WBS 미니 뷰가 표시된다.
2. **Given** 번다운 차트 화면에서, **When** 날짜 범위를 "최근 7일 / 30일 / 사용자 지정"으로 변경하면, **Then** 해당 기간의 잔여 티켓 추이가 갱신된다.
3. **Given** 분석 화면에서, **When** 멤버별 워크로드 탭을 클릭하면, **Then** 각 멤버의 칼럼별(TODO/In Progress/Done) 티켓 수가 스택 바 차트로 표시된다.
4. **Given** WBS 화면에서, **When** 상위 이슈(Goal/Story)를 클릭하면, **Then** 해당 이슈 하위의 티켓 목록과 간트 차트가 확장되어 표시된다.

### Edge Cases

- 워크스페이스 OWNER가 한 명일 때, OWNER 본인을 멤버 목록에서 제거하려 하면 어떻게 처리하는가? → 제거 불가 메시지 표시
- 동시에 두 명이 같은 티켓에 ACTIVE 스프린트를 배정하려 하면? → 마지막 저장 기준 적용 (Last-write-wins)
- 초대 링크를 이미 워크스페이스 멤버인 사용자가 열면? → "이미 멤버입니다" 안내 표시
- 초대 링크를 초대 대상 이메일이 아닌 다른 계정으로 로그인한 사용자가 열면? → "이 초대는 [이메일]로 발송되었습니다. 해당 계정으로 로그인해 주세요." 안내 표시, 수락 차단
- 티켓 담당자가 워크스페이스에서 제거되면 해당 담당자 배정은? → 배정 제거, 티켓은 유지
- 아직 데이터가 없을 때 분석 화면 진입? → "데이터가 충분하지 않습니다. 스프린트를 시작하고 티켓을 만들어보세요." 안내 표시
- ACTIVE 스프린트가 이미 있는 상태에서 두 번째 스프린트를 ACTIVE로 전환하려 하면? → 오류 메시지, 기존 스프린트 완료 후 시작 안내

## Requirements *(mandatory)*

### Functional Requirements

**워크스페이스 (FR-301)**
- **FR-301-01**: 사용자는 이름(1~100자)과 설명(최대 500자)으로 팀 워크스페이스를 생성할 수 있어야 한다. 단, 한 사용자가 OWNER로 소유할 수 있는 팀 워크스페이스는 최대 3개이며, 초과 시 생성이 차단되고 안내 메시지를 표시한다.
- **FR-301-02**: 워크스페이스 생성자는 자동으로 OWNER 역할로 배정되어야 한다.
- **FR-301-03**: OWNER는 워크스페이스 이름과 설명을 수정할 수 있어야 한다.
- **FR-301-04**: OWNER는 워크스페이스를 삭제할 수 있어야 하며, 삭제 전 워크스페이스 이름 재입력으로 의도를 확인해야 한다.
- **FR-301-05**: 워크스페이스 삭제 시 소속된 모든 티켓, 라벨, 댓글, 초대, 스프린트 데이터가 함께 삭제되어야 한다.
- **FR-301-06**: 사용자는 본인이 멤버로 속한 모든 워크스페이스를 전환할 수 있어야 한다.

**멤버 초대 및 RBAC (FR-302)**
- **FR-302-01**: OWNER는 이메일과 역할(MEMBER 또는 VIEWER)을 지정하여 초대를 생성할 수 있어야 한다.
- **FR-302-02**: 초대 시 고유 토큰이 포함된 초대 링크가 생성되어야 하며 유효기간은 7일이다. 토큰은 초대 생성 시 지정한 이메일 주소에 귀속된다.
- **FR-302-03**: 초대 링크를 열고 로그인한 사용자는 초대를 수락하거나 거절할 수 있어야 한다. 단, 로그인된 계정의 이메일이 초대 대상 이메일과 일치하지 않으면 수락이 차단되고 안내 메시지를 표시한다.
- **FR-302-04**: 초대 상태는 PENDING → ACCEPTED / REJECTED / EXPIRED 중 하나로 전이되어야 한다.
- **FR-302-05**: OWNER는 멤버의 역할을 OWNER / MEMBER / VIEWER 간 변경할 수 있어야 한다.
- **FR-302-06**: OWNER는 본인을 제외한 모든 멤버를 워크스페이스에서 제거할 수 있어야 한다.
- **FR-302-07**: VIEWER 역할은 보드 조회와 티켓 상세 조회만 허용되며 생성·수정·삭제는 불가하다.
- **FR-302-08**: MEMBER 역할은 티켓·댓글 CRUD가 가능하나 워크스페이스 설정 및 멤버 관리는 불가하다.
- **FR-302-09**: OWNER 역할은 모든 기능 접근이 가능하다.

**다중 담당자 (FR-303)**
- **FR-303-01**: 티켓당 최대 5명의 담당자를 워크스페이스 멤버 중에서 배정할 수 있어야 한다.
- **FR-303-02**: 담당자 아바타는 티켓 카드에 겹쳐(스택) 표시되어야 하며 최대 3개 아바타 + "+N" 형식을 따른다.
- **FR-303-03**: "내 티켓" 필터 활성화 시 본인이 담당자에 포함된 티켓만 보드에 표시되어야 한다.

**스프린트 관리**
- **FR-SP-01**: OWNER는 스프린트를 이름, 목표, 시작일, 종료일로 생성할 수 있어야 한다.
- **FR-SP-02**: 스프린트 상태는 PLANNED → ACTIVE → COMPLETED / CANCELLED 순으로 전이된다.
- **FR-SP-03**: 한 워크스페이스에 동시에 ACTIVE 상태의 스프린트는 최대 1개여야 한다.
- **FR-SP-04**: 티켓은 특정 스프린트에 배정될 수 있어야 한다.
- **FR-SP-05**: ACTIVE 스프린트를 완료 처리 시 미완료 티켓(BACKLOG/TODO/IN_PROGRESS 상태) 목록을 다이얼로그로 표시하고, OWNER가 각 티켓을 다음 PLANNED 스프린트 또는 백로그(스프린트 미배정) 중 선택하여 이동시켜야 한다. 자동 이동은 허용하지 않는다.

**팀 대시보드 및 분석 (FR-306)**
- **FR-306-01**: 팀 대시보드는 스프린트 현황, Goal 진척율, 멤버별 업무 분포, WBS 미니 뷰를 제공해야 한다.
- **FR-306-02**: 번다운 차트는 선택된 기간 내 잔여량을 일별로 표시해야 한다. 기본 지표는 잔여 티켓 수이며, 스프린트에 스토리 포인트가 입력된 경우 "티켓 수 / 스토리 포인트" 토글을 제공한다.
- **FR-306-03**: 분석 화면은 멤버별 칼럼별 티켓 수(스택 바), 누적 흐름 다이어그램, 벨로시티, 사이클 타임, 라벨 분포를 제공해야 한다.
- **FR-306-04**: WBS 화면은 상위 이슈(Goal/Story) 기준 하위 티켓의 완료율과 간트 형식의 일정을 표시해야 한다.
- **FR-306-05**: 날짜 범위 필터(최근 7일, 30일, 사용자 지정)가 분석 화면과 번다운 화면에 제공되어야 한다.

### Key Entities

- **TeamWorkspace**: 팀 단위의 협업 공간. 이름, 설명, 개설자 정보, 생성일을 가진다. 여러 멤버가 소속될 수 있고 하위에 티켓·라벨·이슈·스프린트를 포함한다.
- **Member**: 워크스페이스에 속한 사용자. OWNER / MEMBER / VIEWER 역할을 가지며, 초대 경로와 합류일이 기록된다.
- **WorkspaceInvite**: 이메일 초대 및 링크 초대를 나타낸다. 고유 토큰, 대상 이메일, 역할, 상태(PENDING/ACCEPTED/REJECTED/EXPIRED), 만료일을 갖는다.
- **Sprint**: 팀 업무 실행 단위. 이름, 목표, 상태(PLANNED/ACTIVE/COMPLETED/CANCELLED), 시작일·종료일, 총 스토리 포인트를 갖는다.
- **TicketAssignee**: 티켓과 멤버 간의 다대다 관계. 하나의 티켓에 최대 5명의 담당자를 연결한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 팀 워크스페이스 생성부터 첫 멤버 초대 링크 발행까지 3단계 이내의 화면 조작으로 완료 가능해야 한다.
- **SC-002**: 초대 링크를 받은 사용자가 링크 클릭 후 워크스페이스 멤버가 되기까지 소요 시간이 1분 이내여야 한다.
- **SC-003**: 역할별 접근 제어(RBAC)가 100% 일관되게 적용되어야 한다 — VIEWER는 어떤 방법으로도 쓰기 작업을 수행할 수 없어야 한다.
- **SC-004**: 팀 대시보드 진입 시 1,000개 티켓, 50명 멤버 규모의 워크스페이스에서도 5초 이내에 화면이 렌더링되어야 한다.
- **SC-005**: 스프린트 상태 전이(시작/완료) 후 팀 대시보드의 통계가 즉시(1초 이내) 반영되어야 한다.
- **SC-006**: 다중 담당자 아바타가 티켓 카드에 정확히 표시되며 "내 티켓" 필터가 담당자 기준으로 정확하게 동작해야 한다 (false-positive 0%).
- **SC-007**: 5개의 팀 화면(대시보드, WBS, 멤버 워크로드, 분석, 번다운)이 모두 접근 가능하고 데이터가 올바르게 시각화되어야 한다.

## Assumptions

- 팀 워크스페이스는 기존 개인 워크스페이스와 독립적으로 운영된다. 사용자는 개인 보드(`/`)와 팀 보드(`/team/[workspaceId]`)를 별도로 사용한다.
- 초대는 이메일 링크 발송 기능 없이 링크 복사 방식으로 시작한다 (이메일 서버 인프라 없음 가정).
- 스위밍 레인(FR-304)과 WIP 제한(FR-305)은 이 버전에 포함되지 않는다 (Board 전면 재작성 필요).
- 분석 데이터는 요청 시점 실시간 집계로 제공된다 (사전 집계 스냅샷 없음). 1,000 티켓 규모에서 실시간 집계가 충분하다고 가정한다.
- 간트/WBS 차트는 외부 차트 라이브러리 없이 순수 SVG로 구현한다 (프로젝트 규칙).
- 스토리 포인트 단위는 정수(1~100)이며 추정 전용이다. 실제 추적이나 번다운 계산에 사용할 수 있다.
- 워크스페이스 멤버십 검증은 팀 API 요청마다 수행된다 (URL 파라미터의 workspaceId 기준).
